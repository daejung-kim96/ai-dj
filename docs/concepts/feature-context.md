# Feature Context

기능 컨텍스트는 어떤 기능을 구현하거나 분석하기 전에 에이전트가 반드시 알아야 하는 정보를 한 묶음으로 정리한 것이다.

`context-router`가 "무엇을 읽을지" 고르는 스킬이라면, `generate-feature-context`는 "읽은 내용을 어떻게 작업 가능한 형태로 정리할지" 다루는 스킬이다.

## 왜 필요한가

AI에게 바로 "이 기능 만들어줘"라고 하면 다음 문제가 생기기 쉽다.

- 요구사항은 읽었지만 API 계약을 놓친다.
- API는 읽었지만 권한 정책을 놓친다.
- 화면은 만들었지만 상태값이나 에러 케이스가 다르다.
- 테스트를 만들었지만 실제 사용자 흐름과 어긋난다.
- 모르는 부분을 질문하지 않고 추측한다.

기능 컨텍스트는 이런 문제를 줄이기 위해 작업 전에 필요한 정보를 한 번 정렬한다.

## 기본 구조

기능 컨텍스트 팩은 보통 다음 내용을 가진다.

- 기능 요약
- 근거 파일과 출처
- 사용자 흐름
- 도메인 객체
- API 표면
- 데이터와 상태 규칙
- UI 영향
- 테스트 시나리오
- 리스크와 열린 질문
- 다음에 사용할 스킬

## 흐름

```text
사용자 요청
-> context-router가 관련 파일을 고름
-> 선택된 파일을 읽음
-> generate-feature-context가 작업 문맥으로 정리함
-> 구현/테스트/리뷰 스킬로 넘김
```

## 좋은 기능 컨텍스트의 조건

좋은 기능 컨텍스트는 짧지만 충분해야 한다.

```text
너무 부족함:
- "리뷰 기능을 만든다."

너무 많음:
- 관련 없는 모든 문서와 코드 요약을 전부 붙인다.

적절함:
- 리뷰 작성 기능의 사용자 목표, API, 상태값, 권한, 테스트 경로,
  확인이 필요한 질문을 근거 파일과 함께 정리한다.
```

## 필수와 추측의 구분

기능 컨텍스트에서 가장 중요한 습관은 "확인한 사실"과 "추측"을 분리하는 것이다.

```text
확인한 사실:
- POST /reviews API는 로그인 사용자가 사용한다.

추측 또는 질문:
- 리뷰 수정은 같은 화면에서 처리하는지 별도 화면이 있는지 확인 필요.
```

AI 에이전트는 빈칸을 자연스럽게 채우려는 경향이 있다. 기능 컨텍스트는 그 빈칸을 추측으로 덮지 않고, 열린 질문으로 드러내게 만든다.

## ai-dj에서의 역할

ai-dj에서 기능 컨텍스트는 구현 직전의 작업 브리프다.

다음 스킬들이 기능 컨텍스트를 입력으로 사용할 수 있다.

- `backend-ddd`
- `backend-eda`
- `frontend-feature`
- `testing-tdd`
- `verify-context-integrity`

## 기억할 문장

기능 컨텍스트는 "AI가 기능을 만들기 전에 알아야 할 것들의 작업용 요약본"이다.

목표는 문서를 많이 요약하는 것이 아니라, 다음 작업자가 바로 판단하고 움직일 수 있게 만드는 것이다.

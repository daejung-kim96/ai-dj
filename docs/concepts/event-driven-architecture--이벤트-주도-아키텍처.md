# Event-Driven Architecture (이벤트 주도 아키텍처)

English name: Event-Driven Architecture, EDA
한국어 이름: 이벤트 주도 아키텍처
쉽게 말하면: 어떤 일이 발생했다는 이벤트를 중심으로 시스템들이 느슨하게 연결되는 구조

## 핵심 생각

EDA는 한 기능이 끝난 뒤 다른 일을 바로 직접 호출하지 않고, "이 일이 발생했다"는 사실을 이벤트로 남기고 다른 쪽이 그 이벤트에 반응하게 만드는 방식이다.

AI 에이전트에게 EDA가 중요한 이유는 비동기 흐름이 눈에 잘 안 보이기 때문이다. 코드는 성공한 것처럼 보여도 이벤트가 유실되거나, 같은 메시지를 두 번 처리하거나, 실패 메시지가 계속 재시도되면서 문제가 생길 수 있다. EDA 스킬은 이런 부분을 설계 단계에서 드러내게 만든다.

## 동기 호출과 이벤트 방식의 차이

리뷰가 작성된 뒤 알림을 보내야 한다고 해보자.

동기 호출 방식:

```text
리뷰 작성 API
-> 리뷰 저장
-> 알림 서비스 직접 호출
-> 알림 생성 완료
-> 사용자에게 응답
```

이 방식은 흐름이 단순하다. 하지만 알림 서비스가 느리거나 실패하면 리뷰 작성 API까지 느려지거나 실패할 수 있다.

이벤트 방식:

```text
리뷰 작성 API
-> 리뷰 저장
-> ReviewCreated 이벤트 기록
-> 사용자에게 응답

나중에:
ReviewCreated 이벤트 소비자
-> 알림 생성
```

이 방식은 리뷰 작성과 알림 생성을 느슨하게 분리한다. 대신 이벤트 발행, 중복 처리, 실패 재시도, 순서 문제를 설계해야 한다.

즉 EDA는 공짜로 복잡도를 없애는 기술이 아니다. 동기 결합을 줄이는 대신 비동기 운영 복잡도를 책임지는 방식이다.

## 주요 개념

| English name | 한국어 이름 | 쉽게 말하면 |
| --- | --- | --- |
| Event-Driven Architecture | 이벤트 주도 아키텍처 | 이벤트를 중심으로 시스템을 연결하는 구조 |
| Domain Event | 도메인 이벤트 | 한 도메인 안에서 실제로 발생한 중요한 사실 |
| Integration Event | 통합 이벤트 | 다른 시스템이나 바운디드 컨텍스트에 알리기 위한 이벤트 계약 |
| Producer | 생산자 | 이벤트를 발행하는 쪽 |
| Consumer | 소비자 | 이벤트를 받아 처리하는 쪽 |
| Topic / Queue | 토픽 / 큐 | 이벤트가 전달되는 통로 |
| Idempotency | 멱등성 | 같은 메시지를 여러 번 처리해도 결과가 망가지지 않는 성질 |
| Outbox Pattern | 아웃박스 패턴 | 상태 변경과 이벤트 기록을 같은 트랜잭션에 저장한 뒤 나중에 발행하는 방식 |
| Retry Policy | 재시도 정책 | 실패한 처리를 언제 어떻게 다시 시도할지 정한 규칙 |
| Dead Letter Queue | 데드 레터 큐 | 계속 처리할 수 없는 메시지를 따로 보내는 곳 |
| Eventual Consistency | 최종적 일관성 | 잠깐 불일치가 있어도 시간이 지나면 맞춰지는 일관성 |

## Domain Event와 Integration Event

이 두 개를 구분하면 설계가 선명해진다.

Domain Event (도메인 이벤트):

```text
ReviewCreated
```

리뷰 도메인 안에서 "리뷰가 생성되었다"는 사실이다. 도메인 내부에서 후속 로직을 연결하는 데 쓸 수 있다.

Integration Event (통합 이벤트):

```text
ReviewCreated.v1
```

다른 바운디드 컨텍스트나 외부 시스템이 구독할 수 있는 계약이다. payload, 버전, 호환성을 더 신중하게 관리해야 한다.

실무에서는 내부 도메인 이벤트를 그대로 외부에 공개하지 않는 편이 안전할 때가 많다. 내부 모델은 바뀔 수 있지만, 외부 이벤트 계약은 쉽게 바꾸면 소비자가 깨지기 때문이다.

## 이벤트 이름은 사실이어야 한다

좋은 이벤트 이름은 이미 일어난 일을 말한다.

```text
ReviewCreated
PaymentCaptured
ExhibitionApproved
NotificationDeliveryFailed
```

반대로 명령처럼 들리는 이름은 피한다.

```text
CreateReview
CapturePayment
ApproveExhibition
SendNotification
```

명령은 "무엇을 해라"이고, 이벤트는 "무엇이 일어났다"이다.

## ai-dj에서 쓰는 방식

ai-dj에서는 `skills/backend-eda`가 비동기 백엔드 흐름을 설계한다.

추천 흐름은 다음과 같다.

```text
context-router
-> generate-feature-context
-> testing-tdd
-> backend-ddd
-> backend-eda
-> verify-context-integrity
```

TDD는 기대 행동을 테스트로 고정하고, DDD는 어떤 도메인 경계에서 일이 발생하는지 정한다. 그 다음 EDA는 그 일이 다른 시스템이나 기능으로 어떻게 안전하게 전달되는지 정한다.

## EDA에서 꼭 봐야 하는 것

EDA를 설계할 때는 이벤트 이름만 정하면 부족하다. 최소한 다음을 같이 봐야 한다.

- 어떤 상태 변화가 이벤트를 발생시키는가
- 누가 이벤트를 발행하는가
- 상태 변경과 이벤트 발행의 트랜잭션 경계는 어디인가
- 이벤트 payload에는 어떤 정보가 들어가는가
- 어떤 정보는 민감해서 빼야 하는가
- 누가 소비하고 어떤 부작용을 만드는가
- 같은 이벤트를 두 번 받으면 어떻게 되는가
- 실패하면 재시도하는가
- 계속 실패하면 어디로 보내는가
- 나중에 다시 처리할 수 있는가

## 예시: 리뷰 작성 후 알림 보내기

요구사항:

```text
사용자가 리뷰를 작성하면, 해당 전시를 관심 등록한 사용자들에게 알림을 보낸다.
리뷰 작성은 알림 발송이 실패해도 성공해야 한다.
알림은 중복으로 생성되면 안 된다.
```

이 요구사항은 EDA 후보가 될 수 있다.

동기 호출로 만들면 알림 실패가 리뷰 작성에 영향을 줄 수 있다. 하지만 요구사항은 리뷰 작성이 알림 실패와 분리되기를 원한다.

EDA 설계는 다음처럼 정리할 수 있다.

```text
Event:
  ReviewCreated

Producer:
  리뷰 생성 유스케이스

Trigger:
  리뷰 저장이 성공했을 때

Payload:
  eventId
  reviewId
  exhibitionId
  authorId
  rating
  occurredAt

Consumer:
  NotificationWorker

Idempotency key:
  eventId

Retry:
  일시적 실패는 재시도

Dead letter:
  payload가 잘못되었거나 계속 실패하면 DLQ로 이동
```

여기서 중요한 판단은 "리뷰 본문 전체를 payload에 넣을 것인가" 같은 질문이다. 알림에 본문 전체가 필요 없다면 보내지 않는 편이 낫다. 이벤트 payload는 필요한 만큼만, 그리고 소비자가 안정적으로 이해할 수 있는 형태로 유지한다.

## Outbox Pattern (아웃박스 패턴)

상태 변경은 성공했는데 이벤트 발행만 실패하면 시스템이 어긋난다.

아웃박스 패턴은 이 문제를 줄이기 위해, 도메인 상태 변경과 이벤트 기록을 같은 트랜잭션에 저장한다. 그 뒤 별도 발행기가 아웃박스에 쌓인 이벤트를 브로커로 보낸다.

```text
1. 리뷰 저장
2. ReviewCreated 이벤트를 outbox 테이블에 저장
3. 트랜잭션 commit
4. outbox publisher가 이벤트 발행
5. 발행 성공 표시
```

아웃박스가 없으면 다음 문제가 생길 수 있다.

```text
1. 리뷰 저장 성공
2. 이벤트 발행 시도
3. 네트워크 오류로 이벤트 발행 실패
4. 사용자는 리뷰를 볼 수 있지만 알림은 영원히 생성되지 않음
```

반대로 이벤트를 먼저 발행하면 더 위험할 수 있다.

```text
1. ReviewCreated 이벤트 발행 성공
2. 리뷰 저장 실패
3. 소비자는 존재하지 않는 리뷰에 대한 알림을 만들려고 함
```

아웃박스 패턴은 이 둘 사이의 간극을 줄인다. 상태 변경과 이벤트 기록을 같은 DB 트랜잭션에 넣고, 실제 브로커 발행은 별도 프로세스가 책임지게 한다.

## Idempotency (멱등성)

이벤트 기반 시스템에서는 같은 메시지가 두 번 올 수 있다고 생각하는 편이 안전하다.

그래서 소비자는 보통 `eventId`, `messageId`, 또는 업무적으로 유일한 키를 이용해 이미 처리한 메시지인지 확인한다.

```text
eventId가 이미 처리됨:
  아무 일도 하지 않고 성공 처리

eventId가 처음 들어옴:
  부작용을 수행하고 처리 기록 저장
```

멱등성이 필요한 이유는 메시지 브로커가 중복 전달을 할 수 있기 때문이다. 네트워크 오류, consumer 재시작, ack 실패 같은 상황에서 같은 메시지가 다시 들어올 수 있다.

알림 예시에서 멱등성이 없으면 이런 일이 생긴다.

```text
ReviewCreated(eventId=abc) 수신
-> 알림 생성
-> 처리 완료 응답 전에 consumer가 죽음
-> broker가 같은 메시지를 다시 전달
-> 알림이 한 번 더 생성됨
```

멱등성을 넣으면 이렇게 처리한다.

```text
ReviewCreated(eventId=abc) 수신
-> processed_events에 abc가 있는지 확인
-> 없으면 알림 생성 + abc 처리 기록 저장

같은 메시지 재수신
-> processed_events에 abc가 있음
-> 아무 부작용 없이 성공 처리
```

멱등성은 "중복 메시지가 절대 오지 않게 한다"가 아니라 "중복 메시지가 와도 망가지지 않게 한다"에 가깝다.

## 실패를 설계해야 한다

EDA에서 실패는 예외 상황이 아니라 기본 고려 대상이다.

| 실패 상황 | 설계 질문 |
| --- | --- |
| 일시적 네트워크 오류 | 몇 번 재시도할 것인가? backoff는 둘 것인가? |
| payload가 잘못됨 | 계속 재시도하지 말고 DLQ로 보낼 것인가? |
| consumer 버그 | 수정 후 DLQ 메시지를 다시 처리할 수 있는가? |
| 중복 메시지 | 어떤 idempotency key로 중복을 막을 것인가? |
| 순서 뒤바뀜 | 순서가 중요한가? 늦게 온 메시지는 어떻게 처리할 것인가? |

이 질문을 미리 하지 않으면, 장애가 났을 때 메시지가 쌓이거나 중복 부작용이 발생한다.

## EDA 테스트는 무엇을 봐야 하나

EDA 테스트는 단순히 "이벤트 발행 함수가 호출됐다"만 보면 부족하다.

좋은 테스트는 다음을 확인한다.

```text
리뷰 생성 성공 시 ReviewCreated가 outbox에 기록된다.
리뷰 생성 실패 시 ReviewCreated가 기록되지 않는다.
같은 eventId를 두 번 소비해도 알림은 한 번만 생성된다.
일시적 실패는 재시도 대상이 된다.
잘못된 payload는 DLQ로 이동한다.
DLQ 메시지를 수정 후 재처리할 수 있다.
```

TDD와 연결하면, EDA도 먼저 실패 테스트로 고정할 수 있다.

## 주의할 점

EDA는 시스템을 느슨하게 연결하지만, 설계를 흐릿하게 만들어도 된다는 뜻은 아니다.

다음은 피한다.

- 이벤트를 명령처럼 이름 짓기
- DB 저장 전에 이벤트부터 발행하기
- 메시지가 절대 중복되지 않는다고 가정하기
- 소비자가 생산자 도메인의 내부 상태를 직접 바꾸기
- 전체 DB row를 payload로 보내기
- 실패 메시지를 무한 재시도하기
- 최종적 일관성을 즉시 일관성처럼 착각하기

## 우리 서비스에서의 활용

아트모아 같은 서비스에 나중에 붙이면, 다음 영역에 특히 유용하다.

- 리뷰 작성 후 알림 발송
- 전시 승인 후 검색 색인 갱신
- 크롤링 데이터 수집 후 정제 작업 시작
- 예약이나 결제 상태 변경 후 후속 처리
- 실패한 알림, 색인, 정제 작업의 재시도와 재처리

공통 스킬에는 특정 서비스의 토픽명이나 브로커 종류를 고정하지 않고, 프로젝트별 이벤트 계약은 `project-profiles/<project>/`에 둔다.

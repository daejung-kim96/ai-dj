# Context Integrity

컨텍스트 무결성은 AI가 만든 결과가 실제로 읽은 근거 파일과 일치하는지 확인하는 개념이다.

`context-router`가 "무엇을 읽을지" 정하고, `generate-feature-context`가 "읽은 내용을 작업 문맥으로 정리"한다면, `verify-context-integrity`는 "그 결과가 근거와 맞는지" 확인한다.

## 왜 필요한가

AI는 그럴듯한 답을 만들 수 있다. 문제는 그 답이 실제 프로젝트 문서나 코드와 어긋날 수 있다는 점이다.

예를 들면 다음과 같은 실수가 생길 수 있다.

- 읽지 않은 파일을 읽은 것처럼 말한다.
- 요구사항에는 없는 기능을 확정된 것처럼 설명한다.
- API 권한이나 상태값을 다르게 적는다.
- 프로젝트별 규칙을 공통 스킬에 섞어버린다.
- 열린 질문이어야 할 내용을 추측으로 채운다.
- 중요한 결정이 생겼는데 문서에 남기지 않는다.

컨텍스트 무결성 검증은 이런 문제를 작업 완료 전에 잡기 위한 단계다.

## 검증 대상

주요 검증 대상은 다음과 같다.

- 라우팅 결과
- 기능 컨텍스트 팩
- 생성된 코드 또는 문서
- 업데이트된 스킬
- 프로젝트 프로필
- 패치노트와 결정 기록

## 기본 질문

검증할 때는 다음 질문을 던진다.

1. 필요한 파일을 실제로 읽었는가?
2. 결과가 읽은 파일의 내용과 맞는가?
3. 확인한 사실과 추측이 분리되어 있는가?
4. 상태값, 권한, API 이름, 도메인 용어가 일관적인가?
5. 프로젝트별 지식이 공통 스킬로 새지 않았는가?
6. 오래 남겨야 할 결정이 기록되었는가?
7. 누락된 컨텍스트나 리스크가 숨겨지지 않았는가?

## pass, warning, fail

검증 결과는 세 단계로 나눈다.

```text
pass:
  근거와 결과가 일치하고, 남은 위험이 작다.

warning:
  작업은 진행 가능하지만 확인해야 할 질문이나 작은 불일치가 있다.

fail:
  결과가 근거와 충돌하거나, 필수 컨텍스트가 없어서 신뢰할 수 없다.
```

## 좋은 검증 리포트

좋은 검증 리포트는 짧고 추적 가능해야 한다.

```text
Status: warning

Findings:
- P1: docs/specs/reviews.md -> 리뷰 중복 정책이 없음 -> 도메인 불변식 확정 전 질문 필요
- P2: api/openapi.yaml -> 401 에러는 명시되어 있으나 403은 없음 -> 권한 실패 케이스 확인 필요
```

중요한 점은 "틀렸다"에서 끝내지 않는 것이다. 어떤 근거에서 어떤 문제가 나왔고, 다음 조치가 무엇인지까지 적어야 한다.

## ai-dj에서의 역할

`verify-context-integrity`는 작업의 브레이크이자 계기판이다.

작업을 막기 위한 스킬이 아니라, AI가 확신하는 척하지 않도록 만드는 스킬이다.

## 기억할 문장

컨텍스트 무결성은 "AI가 말한 내용이 실제로 읽은 근거와 맞는지 확인하는 습관"이다.

AI 하네스에서 이 검증 루프가 있어야, 결과가 점점 신뢰 가능한 작업물로 쌓인다.

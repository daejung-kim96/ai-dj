# Domain-Driven Design (도메인 주도 설계)

English name: Domain-Driven Design, DDD
한국어 이름: 도메인 주도 설계
쉽게 말하면: 비즈니스 규칙과 언어를 코드 구조의 중심에 두는 설계 방식

## 핵심 생각

DDD는 "클래스를 멋지게 나누는 기법"이라기보다, 서비스의 중요한 규칙을 코드 안에서 잘 보이게 만드는 방식이다.

AI 에이전트에게 DDD가 필요한 이유는 분명하다. AI는 데이터베이스 테이블이나 API 모양만 보고 코드를 만들기 쉽다. 하지만 실제 서비스 품질은 "어떤 상태가 가능한가", "누가 무엇을 할 수 있는가", "절대 깨지면 안 되는 규칙은 무엇인가"에 달려 있다. DDD는 그 규칙을 모델의 중심에 둔다.

## DDD는 폴더 구조가 아니다

DDD를 처음 보면 `domain`, `application`, `infrastructure` 같은 폴더를 떠올리기 쉽다. 폴더 구조는 도움이 될 수 있지만, DDD의 핵심은 폴더 이름이 아니다.

더 중요한 질문은 이것이다.

```text
이 서비스에서 중요한 업무 규칙은 무엇인가?
그 규칙을 어떤 말로 부르는가?
그 규칙은 어느 객체나 경계가 책임지는가?
어떤 변경 뒤에도 반드시 지켜져야 하는가?
```

예를 들어 `reviews` 테이블이 있다고 해서 자동으로 `Review` 애그리거트가 되는 것은 아니다. 테이블은 저장 방식이고, 애그리거트는 일관성을 지키는 도메인 경계다.

## 데이터베이스 중심 사고와 도메인 중심 사고

같은 리뷰 기능을 두고도 접근 방식이 다를 수 있다.

데이터베이스 중심 사고:

```text
reviews 테이블이 필요하다.
컬럼은 id, user_id, exhibition_id, rating, content, created_at이다.
POST /reviews API에서 insert하면 된다.
```

도메인 중심 사고:

```text
사용자는 관람하거나 이용한 대상에 대해 리뷰를 남긴다.
한 사용자는 같은 대상에 리뷰를 한 번만 작성할 수 있다.
평점은 1점부터 5점까지만 가능하다.
리뷰 작성자는 자기 리뷰만 수정하거나 삭제할 수 있다.
리뷰가 작성되면 후속 알림이나 통계 갱신이 필요할 수 있다.
```

첫 번째 접근은 저장에는 빠르지만 규칙이 흩어진다. 두 번째 접근은 규칙을 먼저 드러낸다. DDD는 두 번째 접근을 코드 구조에 반영하려는 노력이다.

## 주요 개념

| English name | 한국어 이름 | 쉽게 말하면 |
| --- | --- | --- |
| Bounded Context | 바운디드 컨텍스트 | 같은 단어가 같은 의미로 쓰이는 업무 경계 |
| Ubiquitous Language | 보편 언어 | 기획, 개발, 문서, 코드에서 함께 쓰는 도메인 언어 |
| Entity | 엔티티 | 시간이 지나도 식별자가 중요한 객체 |
| Value Object | 값 객체 | 식별자보다 값 자체가 중요한 불변 객체 |
| Aggregate | 애그리거트 | 함께 일관성을 지켜야 하는 도메인 객체 묶음 |
| Aggregate Root | 애그리거트 루트 | 애그리거트 밖에서 접근하는 대표 객체 |
| Invariant | 불변식 | 어떤 변경 뒤에도 항상 참이어야 하는 규칙 |
| Repository | 리포지토리 | 애그리거트를 저장소에서 가져오고 저장하는 경계 |
| Application Service | 애플리케이션 서비스 | 유스케이스 흐름, 트랜잭션, 저장소 호출을 조율하는 계층 |
| Domain Service | 도메인 서비스 | 한 엔티티에 넣기 애매한 도메인 규칙을 담는 서비스 |

## 예시: 리뷰 도메인 모델링

리뷰 작성 기능을 DDD로 생각하면 다음 순서로 볼 수 있다.

### 1. Bounded Context (바운디드 컨텍스트)

먼저 이 이야기가 어느 업무 경계에 속하는지 본다.

```text
Reviews Context:
  리뷰 작성, 수정, 삭제, 평점, 중복 작성 방지

Exhibitions Context:
  전시 등록, 전시 상태, 전시 공개 여부

Notifications Context:
  알림 생성, 발송, 실패 재시도
```

같은 `status`라는 단어도 컨텍스트마다 뜻이 다를 수 있다. 전시의 `status`는 승인 상태일 수 있고, 알림의 `status`는 발송 상태일 수 있다. 그래서 경계를 먼저 잡는다.

### 2. Ubiquitous Language (보편 언어)

코드에서 어떤 말을 쓸지도 중요하다.

```text
좋은 방향:
  Review
  Rating
  ReviewAuthor
  DuplicateReviewError

애매한 방향:
  Data
  Item
  UserContent
  ErrorCode123
```

보편 언어는 개발자만의 이름이 아니다. 기획서, 대화, 테스트 이름, 코드에서 같은 뜻으로 반복해서 쓰는 말이다.

### 3. Aggregate (애그리거트)

애그리거트는 "같이 저장되는 객체 묶음"이 아니라 "같이 일관성을 지켜야 하는 경계"다.

리뷰 예시에서 중요한 불변식이 다음과 같다고 하자.

```text
한 사용자는 같은 대상에 리뷰를 한 번만 작성할 수 있다.
리뷰 평점은 1점부터 5점까지다.
작성자만 자기 리뷰를 수정할 수 있다.
```

이때 `Review`는 리뷰 자체의 평점, 내용, 작성자 권한 같은 규칙을 책임질 수 있다.

하지만 "같은 사용자가 같은 대상에 리뷰를 이미 썼는지"는 기존 리뷰 목록을 조회해야 한다. 이건 단일 `Review` 객체 혼자 알 수 없다. 그래서 애플리케이션 서비스와 리포지토리 조회, DB unique constraint가 함께 필요할 수 있다.

이 지점이 중요하다. 모든 규칙을 억지로 한 객체 안에 넣는 것이 DDD가 아니다. 규칙이 어느 경계에서 가장 안전하게 지켜지는지 정하는 것이 DDD다.

### 4. Invariant (불변식)

불변식은 "항상 참이어야 하는 규칙"이다.

```text
rating은 1부터 5 사이여야 한다.
deleted 상태의 리뷰는 수정할 수 없다.
작성자가 아닌 사용자는 리뷰를 수정할 수 없다.
```

불변식은 문서에만 있으면 약하다. TDD로 테스트를 만들고, DDD로 이 규칙을 지킬 위치를 정해야 한다.

### 5. Application Service와 Domain Model

둘의 역할을 분리하면 설계가 훨씬 선명해진다.

| 위치 | 책임 |
| --- | --- |
| Application Service | 요청을 받고, 필요한 데이터를 조회하고, 트랜잭션을 열고, 도메인 객체를 호출하고, 저장한다 |
| Domain Model | 평점 범위, 작성자 권한, 상태 전이 같은 비즈니스 규칙을 지킨다 |
| Repository | 애그리거트를 저장소에서 가져오고 저장한다 |
| Infrastructure | DB, ORM, 메시지 브로커, 외부 API 같은 기술 세부사항을 처리한다 |

예시 흐름:

```text
CreateReviewService
  1. userId, exhibitionId, rating, content를 받는다.
  2. 이미 리뷰가 있는지 ReviewRepository로 확인한다.
  3. 중복이면 DuplicateReviewError를 반환한다.
  4. Review.create(...)로 도메인 객체를 만든다.
  5. ReviewRepository.save(review)를 호출한다.
```

여기서 `Review.create`는 평점 범위나 필수 내용 같은 규칙을 지킬 수 있다. 중복 리뷰처럼 저장소 조회가 필요한 규칙은 애플리케이션 서비스와 저장소 경계에서 함께 지킨다.

## ai-dj에서 쓰는 방식

ai-dj에서는 `skills/backend-ddd`가 기능 컨텍스트와 TDD 계획을 받아 백엔드 도메인 모델로 바꾼다.

추천 흐름은 다음과 같다.

```text
context-router
-> generate-feature-context
-> testing-tdd
-> backend-ddd
-> verify-context-integrity
```

TDD가 먼저 "어떤 행동이 맞는가"를 테스트로 고정하고, DDD가 그 행동을 "어떤 도메인 모델이 책임질 것인가"로 바꾼다.

## TDD와 같이 쓰는 이유

DDD에서 가장 중요한 것은 불변식과 상태 전이다.

예를 들어 "한 사용자는 같은 대상에 리뷰를 한 번만 쓸 수 있다"는 규칙이 있다면, 이건 단순한 if문이 아니라 도메인 불변식이다. TDD는 이 규칙을 실패 테스트로 먼저 고정하고, DDD는 이 규칙을 어느 애그리거트나 서비스가 책임질지 결정한다.

```text
TDD: 이 규칙이 깨지면 테스트가 실패하게 만든다.
DDD: 이 규칙을 도메인 모델의 어느 경계에서 지킬지 정한다.
```

둘을 같이 쓰면 AI가 구현을 너무 빨리 시작하지 못하게 하고, 먼저 규칙과 책임을 보게 만든다.

## Aggregate를 고를 때 보는 질문

애그리거트를 고를 때는 관계가 아니라 일관성을 본다.

```text
같이 변경되어야 하는가?
한쪽이 바뀔 때 다른 쪽 규칙이 즉시 깨질 수 있는가?
트랜잭션 안에서 반드시 함께 보호해야 하는가?
외부에서는 어떤 객체를 통해서만 변경하게 할 것인가?
```

단순히 관련이 있다고 하나의 큰 애그리거트로 묶으면 객체가 너무 커지고 수정이 어려워진다. 반대로 너무 잘게 나누면 중요한 불변식이 여러 서비스와 DB 제약에 흩어진다.

초반에는 다음 정도로 생각하면 좋다.

```text
작게 시작한다.
불변식이 깨지는 지점을 본다.
정말 같은 트랜잭션에서 지켜야 하는 규칙만 애그리거트 경계 안에 둔다.
```

## 자주 헷갈리는 구분

| 헷갈리는 것 | 구분하는 법 |
| --- | --- |
| Entity와 Value Object | 식별자가 중요하면 Entity, 값 자체가 중요하면 Value Object |
| Aggregate와 Table | Aggregate는 일관성 경계, Table은 저장 구조 |
| Repository와 DAO | Repository는 도메인 객체 컬렉션처럼 보이고, DAO는 데이터 접근 기술에 더 가깝다 |
| Domain Service와 Application Service | Domain Service는 업무 규칙, Application Service는 유스케이스 조율 |
| Domain Event와 Integration Event | Domain Event는 내부 도메인 사실, Integration Event는 외부에 공개하는 계약 |

## 좋은 DDD 계획의 모양

좋은 DDD 계획은 다음을 분리한다.

- 어떤 바운디드 컨텍스트의 이야기인가
- 어떤 말이 프로젝트에서 쓰는 도메인 언어인가
- 어떤 애그리거트가 일관성을 책임지는가
- 어떤 불변식이 항상 지켜져야 하는가
- 어떤 상태 전이가 가능한가
- 어떤 로직은 도메인이고, 어떤 로직은 애플리케이션 조율인가
- 리포지토리는 어떤 애그리거트를 저장하고 불러오는가
- 어떤 도메인 테스트가 이 규칙을 보호하는가

## 주의할 점

DDD를 한다고 해서 모든 객체를 복잡하게 만들 필요는 없다. 규칙이 거의 없는 단순 CRUD라면 얇게 가는 것이 낫다.

다만 다음은 피한다.

- 데이터베이스 테이블 하나를 무조건 애그리거트 하나로 보기
- 모든 비즈니스 규칙을 서비스 함수에 몰아넣기
- 도메인 객체 안에 ORM, HTTP, 큐 같은 인프라 세부사항 넣기
- 실제 프로젝트 언어와 다른 멋진 용어를 억지로 만들기
- 불변식을 테스트 없이 문서로만 믿기

## 우리 서비스에서의 활용

아트모아 같은 서비스에 나중에 붙이면, 다음 영역에 특히 유용하다.

- 전시, 리뷰, 회원, 알림처럼 규칙이 있는 도메인 경계 나누기
- 리뷰 작성 가능 조건과 중복 작성 방지 같은 불변식 모델링
- 예약, 결제, 승인처럼 상태 전이가 있는 기능 설계
- 크롤링 데이터가 서비스 도메인 데이터로 들어오는 경계 정리
- API 계층과 도메인 규칙이 섞이지 않게 분리

공통 스킬에는 특정 서비스의 사실을 고정하지 않고, 프로젝트별 결정은 `project-profiles/<project>/`에 둔다.

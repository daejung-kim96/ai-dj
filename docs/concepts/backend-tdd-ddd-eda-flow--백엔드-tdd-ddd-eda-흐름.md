# Backend TDD-DDD-EDA Flow (백엔드 TDD-DDD-EDA 흐름)

English name: Backend TDD-DDD-EDA Flow
한국어 이름: 백엔드 TDD-DDD-EDA 흐름
쉽게 말하면: 백엔드 기능을 테스트, 도메인 모델, 이벤트 흐름 순서로 점점 구체화하는 방식

## 한 문장으로 보기

TDD는 "무엇이 맞는 행동인가"를 고정하고, DDD는 "그 행동을 어느 도메인 모델이 책임지는가"를 정하고, EDA는 "그 결과를 다른 기능이나 시스템에 어떻게 안전하게 전달하는가"를 설계한다.

```text
TDD: 행동을 테스트로 고정한다.
DDD: 행동을 책임질 도메인 경계를 정한다.
EDA: 발생한 일을 이벤트로 안전하게 전달한다.
```

세 개는 서로 대체 관계가 아니다. 순서대로 층을 쌓는 관계에 가깝다.

## 예시 요구사항

다음 요구사항 하나로 세 개를 연결해보자.

```text
사용자는 전시에 리뷰를 작성할 수 있다.
한 사용자는 같은 전시에 리뷰를 한 번만 작성할 수 있다.
리뷰가 작성되면 해당 전시를 관심 등록한 사용자에게 알림을 보낸다.
알림 발송이 실패해도 리뷰 작성은 성공해야 한다.
```

이 요구사항에는 세 종류의 고민이 섞여 있다.

| 질문 | 담당 관점 |
| --- | --- |
| 어떤 행동이 맞고 틀린가? | TDD |
| 중복 리뷰 규칙은 어디서 지킬 것인가? | DDD |
| 리뷰 작성 후 알림은 어떻게 안전하게 보낼 것인가? | EDA |

## 1단계: TDD로 행동을 고정한다

먼저 구현하지 않고 테스트 계획을 만든다.

```text
Test case 1:
  given: 사용자가 아직 전시 X에 리뷰를 작성하지 않았다
  when: 사용자가 전시 X에 평점 5점과 내용을 작성한다
  then: 리뷰가 생성된다

Test case 2:
  given: 사용자가 이미 전시 X에 리뷰를 작성했다
  when: 사용자가 전시 X에 다시 리뷰를 작성한다
  then: 중복 리뷰 오류가 발생하고 새 리뷰는 저장되지 않는다

Test case 3:
  given: 리뷰 저장 중 검증 오류가 발생한다
  when: 리뷰 작성이 실패한다
  then: ReviewCreated 이벤트는 기록되지 않는다
```

여기서 TDD의 역할은 구현 방향을 정하는 것이 아니라, 성공과 실패의 기준을 먼저 고정하는 것이다.

좋은 신호:

```text
이 테스트는 아직 구현이 없어서 실패한다.
실패 이유가 요구사항과 연결되어 있다.
테스트 이름만 봐도 비즈니스 행동을 이해할 수 있다.
```

나쁜 신호:

```text
구현을 먼저 다 만든 뒤 그 구현에 맞는 테스트를 생성한다.
테스트가 내부 함수 호출 횟수만 검증한다.
중복 리뷰처럼 중요한 규칙의 실패 케이스가 없다.
```

## 2단계: DDD로 책임을 나눈다

TDD가 행동을 고정했다면, DDD는 그 행동을 책임질 모델을 정한다.

```text
Bounded Context:
  Reviews

Ubiquitous Language:
  Review
  Rating
  ReviewAuthor
  DuplicateReview

Aggregate candidate:
  Review

Invariants:
  rating은 1부터 5 사이여야 한다.
  작성자만 리뷰를 수정할 수 있다.
  같은 사용자는 같은 전시에 리뷰를 한 번만 작성할 수 있다.
```

여기서 모든 규칙이 한 객체 안에 들어가야 하는 것은 아니다.

```text
Review 객체가 직접 지킬 수 있는 규칙:
  rating 범위
  내용 길이
  작성자 권한

Repository나 DB 제약과 함께 지켜야 하는 규칙:
  같은 userId + exhibitionId 조합의 중복 리뷰 방지
```

Application Service는 흐름을 조율한다.

```text
CreateReviewService
  1. 요청을 받는다.
  2. 같은 사용자의 기존 리뷰가 있는지 확인한다.
  3. 없으면 Review.create(...)를 호출한다.
  4. ReviewRepository.save(review)를 호출한다.
  5. ReviewCreated 이벤트 기록을 준비한다.
```

DDD의 역할은 "비즈니스 규칙이 코드 어디에서 책임지는지"를 선명하게 만드는 것이다.

## 3단계: EDA로 후속 처리를 분리한다

리뷰 작성 후 알림 발송은 리뷰 작성 자체와 다른 관심사다. 알림이 실패해도 리뷰 작성은 성공해야 하므로 이벤트 방식이 어울린다.

```text
Event:
  ReviewCreated

Producer:
  CreateReviewService

Transaction boundary:
  Review 저장과 outbox 기록을 같은 트랜잭션으로 처리

Consumer:
  NotificationWorker

Idempotency key:
  eventId
```

이렇게 나누면 리뷰 작성 API는 알림 시스템에 직접 묶이지 않는다.

```text
리뷰 작성 성공
-> ReviewCreated 이벤트 기록
-> API 응답

나중에:
NotificationWorker가 ReviewCreated를 소비
-> 알림 생성
```

EDA의 역할은 "이미 발생한 일을 다른 쪽에 안전하게 전달하는 것"이다.

## 세 개를 같이 쓰면 좋은 이유

TDD만 있으면 테스트는 생기지만, 도메인 책임이 흐릴 수 있다.

DDD만 있으면 모델은 좋아 보이지만, 실제 행동을 깨뜨렸는지 모를 수 있다.

EDA만 있으면 이벤트는 발행되지만, 중복 처리와 실패 복구가 빠질 수 있다.

같이 쓰면 서로를 보완한다.

| 조합 | 얻는 효과 |
| --- | --- |
| TDD + DDD | 불변식과 상태 전이를 테스트로 보호한다 |
| DDD + EDA | 도메인 경계에서 발생한 사실을 이벤트로 표현한다 |
| TDD + EDA | 이벤트 발행, 중복 소비, 실패 처리를 테스트로 고정한다 |
| TDD + DDD + EDA | 행동, 책임, 비동기 전달을 함께 설계한다 |

## AI 에이전트에게 줄 작업 순서

ai-dj에서는 다음 순서가 기본이다.

```text
1. context-router
   필요한 요구사항, 코드, 테스트, 규칙 문서를 고른다.

2. generate-feature-context
   기능의 목표, 상태, 권한, API, 위험을 요약한다.

3. testing-tdd
   성공/실패 행동을 테스트 계획으로 만든다.

4. backend-ddd
   테스트가 보호하는 규칙을 도메인 모델과 경계로 바꾼다.

5. backend-eda
   후속 비동기 처리가 필요하면 이벤트 흐름을 설계한다.

6. verify-context-integrity
   결과가 실제 근거 문서와 맞는지 확인한다.
```

## 초보자가 기억할 기준

처음에는 용어를 완벽히 외우려고 하기보다, 질문을 외우는 편이 낫다.

TDD 질문:

```text
이 기능에서 성공해야 하는 행동은 무엇인가?
실패해야 하는 행동은 무엇인가?
처음에 실패하는 테스트로 표현했는가?
```

DDD 질문:

```text
이 규칙은 어느 도메인 말로 설명되는가?
항상 지켜야 하는 불변식은 무엇인가?
어느 객체나 서비스가 이 규칙을 책임지는가?
```

EDA 질문:

```text
어떤 일이 이미 발생했는가?
누가 그 이벤트를 발행하는가?
누가 소비하는가?
중복, 실패, 재처리는 어떻게 안전하게 처리하는가?
```

이 질문들이 자연스럽게 나오기 시작하면, TDD/DDD/EDA 용어가 조금씩 손에 잡힌다.

# Test-Driven Development (테스트 주도 개발)

English name: Test-Driven Development, TDD
한국어 이름: 테스트 주도 개발
쉽게 말하면: 코드를 먼저 쓰기 전에, 원하는 동작을 테스트로 먼저 고정하는 개발 방식

## 핵심 생각

TDD는 "테스트를 많이 쓰자"가 아니라 "원하는 행동을 먼저 명확히 하자"에 가깝다.

AI 에이전트에게 특히 중요하다. AI는 빠르게 코드를 만들 수 있지만, 무엇이 맞는 행동인지 흐릿하면 그럴듯한 코드를 만들고도 실제 요구사항을 놓칠 수 있다. TDD는 AI가 따라야 할 기준선을 먼저 만든다.

## Red-Green-Refactor (레드-그린-리팩터)

TDD의 기본 반복은 세 단계다.

1. Red (레드): 아직 구현하지 않은 행동을 테스트로 작성하고 실패를 확인한다.
2. Green (그린): 테스트를 통과시키는 가장 작은 구현을 한다.
3. Refactor (리팩터): 테스트가 통과하는 상태에서 구조를 정리한다.

중요한 점은 실패를 먼저 보는 것이다. 실패하지 않는 테스트는 새 행동을 검증하고 있는지 알기 어렵다.

## TDD가 헷갈리는 이유

TDD를 처음 보면 "테스트 코드를 먼저 쓰는 것"처럼 보인다. 맞는 말이지만, 그것만으로는 부족하다.

TDD의 핵심은 다음 순서를 강제로 만드는 데 있다.

```text
무엇이 맞는 행동인지 정한다
-> 그 행동을 테스트로 표현한다
-> 아직 구현이 없어서 실패하는지 확인한다
-> 통과할 만큼만 구현한다
-> 테스트가 지켜주는 상태에서 구조를 정리한다
```

그래서 TDD는 단순히 테스트 파일을 많이 만드는 일이 아니다.

| 헷갈리는 표현 | 실제 의미 |
| --- | --- |
| 테스트가 있다 | 코드 실행 결과를 확인하는 장치가 있다 |
| 테스트 커버리지가 높다 | 많은 코드 줄이 테스트 중 실행된다 |
| TDD를 한다 | 원하는 행동을 실패 테스트로 먼저 고정하고 구현을 진행한다 |

커버리지가 높아도 중요한 행동을 검증하지 않으면 TDD라고 보기 어렵다. 반대로 테스트 수가 적어도 핵심 규칙을 정확히 고정하면 좋은 TDD가 될 수 있다.

## ai-dj에서 쓰는 방식

ai-dj에서는 `skills/testing-tdd`가 요구사항을 테스트 계획으로 바꾼다.

보통 흐름은 다음과 같다.

```text
context-router
-> generate-feature-context
-> testing-tdd
-> 구현 스킬
-> verify-context-integrity
```

`testing-tdd`는 다음을 먼저 정리한다.

- 어떤 행동을 보호하려는가
- 그 행동의 근거 문서는 무엇인가
- 성공 조건은 무엇인가
- 실패 조건은 무엇인가
- 권한, 상태 변화, 이벤트, 에러는 무엇인가
- 어떤 레벨의 테스트가 가장 적절한가

## 테스트 레벨

모든 행동을 E2E 테스트로 검증할 필요는 없다. 가능한 한 작고 빠른 테스트로 같은 확신을 얻는 것이 좋다.

| English name | 한국어 이름 | 언제 쓰나 |
| --- | --- | --- |
| Unit Test | 단위 테스트 | 순수 함수, 계산, 포맷팅, 작은 검증 로직 |
| Domain Test | 도메인 테스트 | 불변식, 상태 전이, 정책, 권한 규칙 |
| API Contract Test | API 계약 테스트 | 요청/응답 형식, 상태 코드, 에러 형식 |
| Integration Test | 통합 테스트 | DB, 큐, 외부 어댑터, 여러 모듈의 연결 |
| E2E Test | 엔드투엔드 테스트 | 실제 사용자 흐름 전체 |

## 예시: 리뷰 중복 작성 방지

예를 들어 다음 요구사항이 있다고 하자.

```text
사용자는 같은 대상에 리뷰를 한 번만 작성할 수 있다.
이미 리뷰를 작성한 사용자가 다시 리뷰를 작성하면 중복 리뷰 오류가 발생해야 한다.
```

초보자가 바로 구현부터 시작하면 보통 이런 흐름이 된다.

```text
컨트롤러 만든다
서비스 만든다
DB에 저장한다
중복이면 적당히 if문을 추가한다
나중에 테스트를 붙인다
```

이 방식은 빠르게 보이지만, 중복 기준이 무엇인지 흐려지기 쉽다.

- 같은 사용자인가?
- 같은 전시인가?
- 삭제된 리뷰도 중복에 포함하는가?
- 임시 저장 리뷰도 중복인가?
- 오류는 어떤 형태로 반환하는가?

TDD는 먼저 질문을 테스트로 바꾼다.

```text
테스트 이름:
  같은 사용자가 같은 대상에 두 번째 리뷰를 작성하면 거절된다

given:
  사용자 A가 전시 X에 이미 리뷰를 작성했다

when:
  사용자 A가 전시 X에 다시 리뷰를 작성한다

then:
  중복 리뷰 오류가 발생한다
  새 리뷰는 저장되지 않는다
```

이 테스트는 아직 구현이 없으므로 실패해야 한다. 여기서 실패는 좋은 신호다. "우리가 원하는 행동을 아직 코드가 만족하지 못한다"는 사실을 보여주기 때문이다.

그 다음 구현은 가장 작게 한다.

```text
1. 리뷰 저장 전에 같은 userId + targetId 리뷰가 있는지 확인한다.
2. 이미 있으면 DuplicateReviewError를 반환한다.
3. 없으면 리뷰를 저장한다.
```

테스트가 통과하면 그때 구조를 정리한다.

```text
중복 확인 로직을 어디에 둘까?
- 도메인 정책인가?
- 애플리케이션 서비스의 중복 조회인가?
- DB unique constraint도 같이 필요한가?
```

여기서 DDD와 연결된다. TDD가 "이 행동이 반드시 필요하다"를 고정하고, DDD가 "이 규칙을 어느 도메인 경계에서 책임질지"를 정한다.

## 좋은 테스트와 약한 테스트

좋은 TDD 테스트는 구현 방법보다 사용자가 기대하는 행동을 말한다.

약한 테스트:

```text
ReviewService.createReview가 reviewRepository.findByUserId를 한 번 호출한다
```

이 테스트는 내부 구현에 너무 묶여 있다. 나중에 repository 메서드 이름이 바뀌거나 조회 방식이 바뀌면, 행동은 그대로인데 테스트가 깨질 수 있다.

더 나은 테스트:

```text
이미 리뷰를 작성한 사용자는 같은 대상에 두 번째 리뷰를 작성할 수 없다
```

이 테스트는 구현이 바뀌어도 중요한 비즈니스 규칙을 계속 보호한다.

## AI에게 TDD를 시킬 때 중요한 점

AI에게 "테스트도 만들어줘"라고만 말하면 구현을 먼저 상상하고 그 구현에 맞춘 테스트를 만들 수 있다.

더 좋은 요청은 다음처럼 행동과 실패 이유를 먼저 요구하는 것이다.

```text
이 요구사항을 바로 구현하지 말고 TDD 계획부터 만들어줘.
각 테스트는 given/when/then으로 쓰고,
왜 처음에는 실패해야 하는지도 적어줘.
구현 파일은 테스트 계획을 확인한 뒤에 바꿔줘.
```

ai-dj의 `testing-tdd` 스킬은 바로 이 흐름을 강제하기 위한 장치다.

## 좋은 TDD 문서의 모양

좋은 TDD 계획은 구현 세부사항보다 행동을 말한다.

```text
given: 사용자가 이미 같은 전시에 리뷰를 작성했다
when: 같은 전시에 다시 리뷰를 등록한다
then: 중복 리뷰 오류가 발생하고 새 리뷰는 저장되지 않는다
source: 리뷰 요구사항 문서
```

이런 식으로 쓰면 백엔드, 프론트엔드, API 중 어디를 구현하더라도 같은 행동 기준을 공유할 수 있다.

## 주의할 점

TDD는 모든 것을 느리게 만드는 절차가 아니다. 오히려 AI와 함께 일할 때는 되돌아가는 시간을 줄인다.

다만 다음은 피한다.

- 구현을 먼저 끝내고 테스트만 나중에 붙이기
- 내부 함수 이름이나 구조에 지나치게 묶인 테스트 만들기
- 작은 도메인 테스트로 충분한데 무거운 E2E 테스트만 만들기
- 실패 이유를 확인하지 않고 테스트가 있다고 안심하기

## 우리 서비스에서의 활용

아트모아 같은 서비스에 나중에 붙이면, 다음 영역에 특히 유용하다.

- 전시 리뷰 작성, 수정, 삭제 규칙
- 회원 권한과 소유권 규칙
- 예약, 결제, 알림처럼 상태 전이가 있는 기능
- API 응답 형식과 에러 코드 유지
- 크롤링 데이터 정제 규칙의 회귀 방지

공통 스킬에는 특정 서비스 이름을 고정하지 않고, 프로젝트별 사실은 `project-profiles/<project>/`에 둔다.
